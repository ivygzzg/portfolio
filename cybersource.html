Case Study · Security · UX/UI

Integrating Cybersource DM & 3DS — a bridge between security and usability

Project Cybersource · Velpay
Role UX/UI Designer
Team Engineer team · Scrum Master
Scope 1 month

Context

What is Cybersource?

Velpay sought to strengthen fraud prevention in its digital transactions by integrating Cybersource Decision Manager (DM). The challenge was not just the technical API connection, but incorporating new Backoffice features so internal teams (Commercial, Operations, Risk) could configure the tool flexibly for each merchant.

I redesigned the interface and worked closely with the engineering team to improve the UX/UI of the Backoffice, the merchant-facing Assistant portal, and the end-user checkout experience.

What is Cybersource Decision Manager?

A fraud management platform using AI and configurable rules to decide whether a transaction is accepted, rejected, or sent for review in milliseconds. Real-time evaluation, custom fraud rules, full audit logs, and compliance with international standards.

Problem

Fraud prevention was manual, slow, and opaque for merchants

High fraud incidence in card-not-present transactions

Manual review processes caused delays of up to 24 hours. CNP fraud rate was at 1.2% of monthly transactions.

No granular Backoffice configuration

Some merchants had their own fraud validators but the BO had no module to manage this logic, limiting autonomy and personalization.

False positives damaging user experience

~4.5% false positive rate was rejecting legitimate purchases, frustrating users and generating constant support calls.

No merchant transparency

Merchants had no visibility into Risk Scores or rejection reasons, creating distrust and dependence on Velpay for every change.

Objective

Design secure, easy-to-use flows that give merchants autonomy and internal teams full control

Design intuitive flows for enabling DM and 3DS. Allow merchants to manage Cybersource keys and services independently without depending on Velpay for every change.

Give internal teams clear administrative control. Improve transparency by surfacing Risk Score and rejection reasons. Ensure the checkout experience is smooth and aligned with international security standards.

Research & Insights

Stakeholder interviews revealed competing needs across teams

Benchmarks from Stripe Radar and Adyen RevenueProtect confirmed: transparency toward merchants is a competitive advantage. Native fraud dashboards with granular visibility build trust and reduce friction.

Key Learning

The real challenge was designing for two distinct audiences with separate but coherent flows, while making complex security configuration feel simple, guided, and trustworthy.

Solution

Three integrated touchpoints solving a different layer of the problem

A — Backoffice

New module in Merchant Configuration

  • Activate/deactivate DM and 3DS per merchant
  • Configure transaction type: processed or successful
  • Default: fraud tool enabled (Processed: ON)
  • Clear hierarchical navigation in Other Services
  • Visual alignment of radio buttons and labels

B — Assistant (Merchants)

Module to load and manage Cybersource keys

  • Manual entry of public and secret keys
  • Upload .pem file method
  • Edit or delete existing keys
  • No key loaded: DM and 3DS unavailable
  • Valid key: services auto-enabled with ON/OFF toggle

C — Checkout (End users)

3DS auth + real-time DM evaluation

  • 3DS challenge via iframe when required
  • Visible trust logos and clear messaging
  • DM evaluates risk in real time
  • Merchants see Risk Score and rejection reason

Impact

Autonomy, transparency, and control at every level

  • Commercial: activate/deactivate DM per client without engineering
  • Merchants: use their own rules engine or Velpay's
  • Internal users: cleaner BO with visually aligned options

Ideation — User Flows

Three flows for three audiences

Designing for internal users and external merchants required separate but coherent flows. The checkout flow depended entirely on configurations set by the other two — getting the hierarchy right was essential.

Every design decision in the BO had a downstream effect on what merchants could configure, and what their customers experienced at checkout. This interdependency shaped every interaction pattern.

01
Backoffice — Internal User

Login → Merchant Config → Other Services → API → Cybersource → Fraud Tool. Review default state (Processed: ON). Adjust type or toggle DM. Changes saved to DB. API validates if service is contracted.

02
Assistant — Merchant Admin

Login → Settings → Cybersource Keys. System shows current state. Options: manual entry, .pem upload, edit/delete. No key: DM/3DS unavailable. Valid key: services auto-enabled. Independent ON/OFF switch per service.

03
Checkout — End User

Initiates checkout → enters payment data → DM evaluates risk in real time. If 3DS required: iframe challenge → OTP or app auth. Payment processed. Merchant receives Risk Score and rejection reason in report.

Design Execution

Making complex configuration feel simple and trustworthy

Backoffice — Other Services module

The new Merchant Configuration section introduces a clean hierarchy. The Fraud Tool checkbox is visible by default (Processed: ON), with clear radio button labels. Visual alignment was a priority — a misaligned label in a critical payment configuration creates real errors.

Assistant — Key management

Merchants load Cybersource keys manually or via .pem. Credentials are shown once and encrypted thereafter. A warning modal prevents accidental loss. The ON/OFF toggle only activates when a valid key is loaded, preventing configuration errors.

Before

100% manual fraud review — delays up to 24 hours
4.5% false positive rate rejecting legitimate transactions
No BO module to configure fraud rules per merchant
No Risk Score or rejection reason visible to merchants

After

Real-time validation in milliseconds with Cybersource DM
Estimated false positive reduction to 1.8% in phase one
BO with antifraude config module — autonomy for all teams
Merchants manage their own keys — fewer internal dependencies

Learnings

Security UX is about clarity, not complexity

Designing for two distinct audiences requires separate but coherent flows. Security configuration must be unambiguous — what is technically complex (keys, fraud rules) must translate into simple, guided steps.

Visual consistency is not cosmetic when users are configuring critical financial settings. A misaligned radio button can lead to real errors with real business impact.

Transparency toward merchants generates trust and reduces the support burden on internal teams. In a payments ecosystem, good UX makes security feel like added value — not friction.

This project shows how UX can be the bridge between security and usability — making the complex feel simple, and the invisible feel trustworthy.

Integrating Cybersource DM and 3DS in Velpay reduced fraud and costs, improved checkout for end users, gave merchants autonomy and control, and made the Backoffice genuinely usable for the people who rely on it every day.
